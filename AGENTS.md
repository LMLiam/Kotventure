# AGENTS.md

Use this guide when you change this repository with an AI coding agent.
Human contributors must read [`CONTRIBUTING.md`](.github/CONTRIBUTING.md). This guide gives the same rules in a format for agents.

> **Priority:** An explicit maintainer instruction has first priority. This guide has second priority. Agent defaults have last priority.

---

## 1. Project purpose

Kotventure is a multiplatform Kotlin DSL for [Adventure](https://github.com/PaperMC/adventure). It supports Paper, Velocity, and Fabric.
It adds typed MiniMessage validation, component tests, ANSI preview, and code generation.

Read these sources before a non-trivial change:

- [`docs/DESIGN.md`](docs/DESIGN.md) defines the architecture, module map, DSL surface, and roadmap. It controls design decisions.
- [`docs/ROADMAP.md`](docs/ROADMAP.md) defines the phase sequence.
- The roadmap epic and its milestone issues on GitHub define the work items.

## 2. Primary rules

1. Use one issue for one small vertical change and one small PR.
2. If a change becomes large or has two purposes, stop. Divide it into smaller changes.
3. Wrap `net.kyori.*` types. Do not make a new implementation of an Adventure feature.
4. Obey the architecture in section 4. Do not add a dependency that the design does not specify.
5. Add tests in the same change.
6. Implement only the requirements in the issue. Do not add speculative abstractions.

## 3. Build, test, and format

The build uses JDK **25** through the Gradle toolchain. Always use the Gradle wrapper.

```bash
./gradlew build          # compile + test + lint + verify coverage
./gradlew test           # Kotest suites
./gradlew ktlintFormat    # auto-fix Kotlin style   (or: ./gradlew spotlessApply)
./gradlew ktlintCheck spotlessCheck   # verify style
./gradlew koverHtmlReport   # aggregated coverage report → build/reports/kover/html/index.html
```

Run the format task and applicable tests before you commit. All CI checks must pass.

For the CI architecture, refer to [`docs/CI.md`](docs/CI.md).

Do not add comments that narrate workflows, scripts, or Gradle and CI configuration. Do not add comments such as these examples:

- "why this exists"
- "configuration-cache safe"
- "lazy because..."

Use job names, step names, task `description` values, and `docs/CI.md` to give the intent.
You can add an action version after a SHA, for example `# vX.Y.Z`.
Do not put internal audit IDs in repository files or public documents. Do not put deferred work in `docs/`.

Release-please generates `CHANGELOG.md`. Do not edit this file.
Ignore Qodana StringSpec format alerts and the neutral Qodana for JVM check. CodeRabbit can stop if it has no credits.

`koverVerify` runs with `check` and `build`. It fails if total line coverage is less than 85%.
[`gradle/coverage.gradle`](gradle/coverage.gradle) defines this baseline. Do not decrease it. Add tests instead.
For the threshold policy, refer to [`CONTRIBUTING.md`](.github/CONTRIBUTING.md).

[`gradle/libs.versions.toml`](gradle/libs.versions.toml) contains shared dependency and plugin coordinates.
Add a shared version there first. Then, use a catalogue alias or an applicable Gradle helper script.

## 4. Architecture requirements

For the complete architecture, refer to section 4 of `docs/DESIGN.md`.

- Add a module only in its planned phase. Put each module in `modules/<name>` and enable it in `settings.gradle`.
- The `core` module can depend only on `adventure-api`. Do not add MiniMessage, coroutine, or platform code to `core`.
- Use hybrid extensibility only if a feature needs runtime lookup.
- The feature that owns the lookup must provide a small explicit registry, such as `ThemeRegistry`.
- Do not use a hidden process-global registry, `ServiceLoader`, SPI, reflection, or a classpath scan.
- Do not create one ambient registry. Do not restore the former `AdventureDsl` object.
- `explicitApi()` applies to library modules. Add explicit visibility, an explicit return type, and KDoc to each public or protected declaration.
- Use this KDoc link priority: `[Ref]`, `[text][fully.qualified.name]`, and then inline `` `code` ``.

## 5. Code quality and structure

### Single responsibility

- Give each file, class, or object one reason to change. Divide a file that has two purposes.
- Put only one top-level class, interface, or object in each file. Give the file the same name.
- Put a scope and its builder in different files. Put a base type and its subtype in different files.
- Top-level functions and properties for one feature can share a file, such as `NamedColors.kt`.
- A violation of this file rule requires rework.
- Prefer many small, focused files. Divide a file that is difficult to understand as one unit.
- Give each function one task at one abstraction level. Extract code if the function becomes complex or deeply nested.

### Package structure

- Organise packages by feature. Use packages such as `core.text`, `core.style`, `core.color`, and `minimessage.template`.
- Do not organise packages as `builders`, `models`, `utils`, or `helpers`.
- Put public DSL entry points in the feature package. Keep implementation details `internal`.
- Do not create a `util`, `misc`, or `common` package. Put each item with its feature.

### API shape

Before you plan a public API, read "Designing the API shape" in the `idiomatic-kotlin-dsl` skill.

- Prefer compile-time checks to runtime checks. Use a property for a named value when possible.
- Consider a delegated property before a typed key or string lookup. It gives a compile-checked name for registry interoperation.
- Treat a runtime check for a missing, incorrect, or duplicate name as an API design defect.
- Give each use case one API. Do not provide typed and string overloads for the same operation.
- Provide one explicit string bridge only when configuration, MiniMessage, or cross-plugin interoperation needs it.
- Do not register a value during construction. Register it with an explicit call to the registry that owns it.
- Use the smallest public surface that satisfies the acceptance criteria. Add convenience operations only when a requirement needs them.
- Reject malformed input. Do not correct it automatically.
- Throw `IllegalStateException` for a duplicate singleton argument or mixed polarity. Name the applicable argument.
- Keep repeated arguments in call order. Do not use last-write-wins behaviour.
- Model parser capabilities in each scope. Do not model the semantic intent of a selector name.
- Treat issue and design snippets as examples. Design a type-safe API that gives the intended result.
- A prior decision does not prevent a better design. Show the maintainer concrete call sites before you use a different design.

### Idiomatic Kotlin requirements

- Build DSLs with receiver lambdas and `@DslMarker`.
- Prefer `val`, immutable data, data classes, sealed types, and enums.
- Use extension functions for the DSL surface. Use expression bodies for one-line functions.
- Do not use `!!`. Prefer `?:`, `?.`, `requireNotNull`, or `checkNotNull`.
- Use scope functions only when they make the code easier to read.
- Use clear, complete names. Do not use abbreviations or Hungarian notation.
- Use new Kotlin features when they make the intent clearer. Ask the maintainer before you use an experimental feature.
- Explicit backing fields are enabled. Prefer them to a separate property with an underscore prefix.
- Prefer `inline fun <reified T>` to a `Class<*>` or `KClass` parameter.
- Put scope-specific operations in the DSL scope. Prefer `listOf` or an operator to a prefixed global name.

Do not use these patterns:

- Java-style getters or setters when a property is sufficient.
- A manual builder when a DSL is sufficient.
- `java.util.*` when the Kotlin standard library has the applicable type.
- Wildcard imports.
- God objects or static utility classes.
- Public mutable state across a module boundary.
- An interface with one implementation without a current requirement.
- Premature generic abstractions.

### Adventure wrappers

- Each builder must make a valid Adventure object. Construct the real `net.kyori` type in a test and check its value.
- Read the `adventure-reference` skill before you use an Adventure API that you do not know. Do not guess its API.

### Quality measures

Use these structural limits:

- Keep cyclomatic complexity at 10 or less for each function.
- Extract named functions or predicates instead of nested conditions.
- Reduce the number of different operators and operands in one function. Divide long or complex expressions.
- Test all behaviour through the public surface. Do not test an internal seam to compensate for a poor design.
- Treat the Kover value as a minimum threshold. Tests must check behaviour, not lines.

Apply these review criteria:

- **Readability:** Use full names, small functions, and one abstraction level.
- **Maintainability:** Put each policy in one place. Minimise dependencies between features.
- **Portability:** Use small functions with few dependencies. Keep `core` dependent only on `adventure-api`.
- **Extensibility:** Use sealed types or enums for closed sets. Do not make extension depend on a god object.
- **Reliability:** Reject malformed input. Do not add allocation or repeated scans without a need.
- **Efficiency:** Do not reduce clarity for an optimisation without a measurement.
- **Documentation:** Add KDoc to each public API. State contracts, invariants, and exceptions instead of the signature.

## 6. Tests

- Use Kotest for all tests. Add tests with each behavioural change. Write the test first when practical.
- Use the project matchers from the `test` module. Refer to the `writing-component-tests` skill.
- Use Kotventure DSL entry points in test setup, samples, and documents when an equivalent exists.
- For example, use `audienceOf(...)` instead of `Audience.audience(...)`. Use `emptyAudience()` instead of `Audience.empty()`.
- Use raw `net.kyori` values for expected assertion values. This verifies the DSL against Adventure, not against itself.
- Use snapshot tests for applicable message regressions.

## 7. Commits, pull requests, and branches

- Use `verb(area): something` for the PR title and each commit subject. Use lower-case letters for the verb and area.
- The PR workflow validates PR titles and commit subjects. The CI workflow validates commits pushed to `master`.
- Refer to `CONTRIBUTING.md` for recommended verbs and areas.
- Use `type/issue-<n>/short-desc` for a branch, for example `feat/issue-19/style-dsl`.
- Put `Closes #<n>` in the PR. Select the PR template for the change type.
- If an issue is in a GitHub Project, add the PR to the same project.
- Copy `Status`, `Priority`, `Area`, `Kind`, `Effort`, `Risk`, and `Contributor fit` from the issue.
- Run `gh project item-list` to verify the project fields before you report completion.

## 8. Definition of done

- [ ] The correct module and feature package contain the public DSL and its implementation.
- [ ] Matcher and snapshot tests pass.
- [ ] `ktlintCheck` and `spotlessCheck` pass.
- [ ] Public API has KDoc and satisfies `explicitApi()`.
- [ ] User-facing changes include applicable changes in `docs/`.
- [ ] `CHANGELOG.md` has no manual changes.
- [ ] The title and commit subjects have the required format.
- [ ] The PR links its issue.

## 10. Protected areas

- Keep build configuration and module-layout changes small and related to the issue.
- Do not change the public API of a frozen module without a binary-compatibility baseline update.

## 11. How to write to the maintainer

The maintainer has ADHD. Shape each reply so that it is possible to act on immediately.
If you have the `i-have-adhd` skill, use it. These rules apply with or without that skill.

- Start with the action. Put the command, the path, or the code first. Give the background after it, or not at all.
- Do not use an opening sentence that announces your plan ("Let me…", "I'll…", "Great question").
- Do not use a closing sentence that repeats the work or asks "anything else?".
- Number a task that has more than one step. Give one action in each step.
- Restate the position in a sequence each time, for example "Step 3 of 5 done".
- Give a time estimate in concrete units, for example "about 15 minutes", not "some work".
- Keep a list to five items. If it becomes longer, divide it into "do now" and "later".
- Report an error with its cause and its correction. Do not use "Uh oh" or "There seems to be a problem".
- Show what now operates, with the command that shows it.
- Finish one subject before you introduce a second one. Offer the second subject as a separate question.
- End with one action that takes less than two minutes.

Give a full explanation when the maintainer asks you to explain or to review a design. Add headings so that
the text is easy to scan. The rules about the first sentence and the last sentence continue to apply.
